# opensignal-job-intel

`opensignal-job-intel` is a lightweight Python-based job discovery and qualification system for human-in-the-loop job search workflows.

## Goal

Build a pragmatic foundation for:

- capturing jobs manually first
- normalizing them into a canonical schema
- storing them in SQLite
- preparing for future qualification with rules and LLM reasoning
- keeping the architecture open for later source integrations such as LinkedIn, Otta, Wellfound, and Indeed

## Purpose

This project is intended to help a candidate review and prioritize job opportunities without automatic application or autonomous outreach.

## Current Status

The repository now contains the first implemented OpenSpec change, `v1-linkedin-ingestion-storage`. The current repo state is a working local-first prototype: it can load a professional compass, ingest fixture-backed LinkedIn job data through the source adapter boundary, store canonical jobs in SQLite, and produce a first-pass rule-based evaluation for human review.

- `profiles/`: local professional compass input plus a committed template
- `tests/`: `unittest` coverage for normalization and SQLite persistence behavior

## Constraints

- Python implementation
- object-oriented first design
- human-in-the-loop only
- no auto-apply
- no autonomous outreach
- SQLite preferred for v1

## Current V1 Implementation

The first increment implements:

- a canonical `JobRecord` model for normalized postings
- a stored professional compass profile used as the user-facing input
- a committed professional compass template for repository users
- a source adapter boundary via `JobSourceAdapter`
- a minimal LinkedIn adapter backed by local JSON fixtures
- a SQLite repository with duplicate-safe upsert behavior and additive schema initialization for new canonical fields
- a lightweight evaluator that classifies and scores each job against the compass
- a CLI workflow to ingest and score stored jobs locally

This keeps LinkedIn ingestion isolated from any browser automation, scraping stack, or official API dependency while still making the ingestion boundary testable.

## First-Run Workflow

The primary user input is the stored professional compass at:

```text
profiles/professional_compass.json
```

Start by copying the template:

```bash
cp profiles/professional_compass.template.json profiles/professional_compass.json
```

For the current v1 stub adapter, create a small LinkedIn-style source fixture:

```json
[
  {
    "id": "123",
    "company": "Example Corp",
    "title": "Staff Data Architect",
    "description": "Remote product team building modern data and AI systems with Python, SQL, Snowflake and LLM workflows.",
    "posted_at": "2026-04-15T12:00:00Z",
    "salary": "$7,000 - $10,000 monthly",
    "link": "https://www.linkedin.com/jobs/view/123/"
  }
]
```

Run ingestion with `python3`:

```bash
python3 main.py ingest-linkedin --compass-file profiles/professional_compass.json --source-file /path/to/linkedin_jobs.json
```

Optional flags:

- `--db-path data/jobs.db` to choose the SQLite file location
- `--limit 10` to control how many stored jobs are printed after ingestion

## LinkedIn Adapter Assumption

The v1 LinkedIn adapter still reads a local JSON file that represents already-collected LinkedIn job records. That file is now explicitly adapter-side fixture input, not the user's professional search profile. The user-facing input is the professional compass JSON. This keeps the ingestion boundary correct without prematurely choosing scraping, browser automation, or partner-gated API integration.
